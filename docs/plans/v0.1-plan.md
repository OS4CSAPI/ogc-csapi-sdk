# OGC CSAPI SDK — v0.1 Plan

**Status:** Draft  
**Date:** 2026-05-08  
**Owner:** OS4CSAPI / Sam Bolling (interim)  
**Audience:** OS4CSAPI org maintainers, code-sprint participants, prospective enterprise adopters

## 0. One-sentence mission

> Enable a technical team to take one existing sensor data source, map it to CSAPI, publish observations through a CSAPI server, validate the endpoint, and inspect the result visually — in one afternoon.

If v0.1 cannot deliver that in one afternoon on a clean laptop, it has failed.

## 1. Product framing

- **External name:** `OGC CSAPI SDK`
- **Subtitle:** *Sensor System Integration Toolkit for OGC API – Connected Systems*
- **Internal architecture truth:** standards-based integration toolkit, not a language-binding SDK.
- **Posture:** implementation-neutral. CS GO is the v0.1 golden path because it is the cleanest reference today; the SDK does not declare CS GO as the only path.

## 2. Non-goals for v0.1

Explicitly excluded — surfaced here so reviewers stop asking:

- Generated language clients (Python/TypeScript/Java/.NET SDK packages).
- Full Part 2 command/tasking write paths beyond a read-only example.
- Part 3 (Pub/Sub) — not yet certifiable; OSH-addons branch is experimental.
- Part 5 (Binary encodings via SWE Common 3.0) — `tipatterson-dev/SWECommonBinaryEncodings` is AI-generated, single-author, not authoritative.
- PKI wizard / production hardening / FIPS validation (separate v0.2 defense track).
- Multi-server parity matrix beyond the three documented servers.
- Authoring SensorML by hand. v0.1 uses minimal-viable SensorML JSON only.

## 3. Audience for v0.1

Primary: an integrator at an organization that **already has a sensor system and a data feed**, has been told "make it CSAPI-compatible," and has never seen CSAPI before.

Secondary: code-sprint participants who need a runnable reference stack.

Not v0.1: defense / IC contractors with air-gap, FIPS, RHEL constraints. They need v0.2 defense track. v0.1 should be honest about this in `START-HERE.md`.

## 5. The five anchor artifacts (must ship in v0.1)

If any one of these slips, v0.1 ships without it rather than waiting:

### A. CSAPI Mapping Workbook (`worksheets/csapi-mapping-workbook.md`)

The single most differentiating artifact. Walks an integrator through:

1. What real-world systems do you have? (sensor / platform / actuator / process / aggregate)
2. What deployments? (fixed / mobile / hierarchical / time-bounded)
3. What features of interest? (area / object / platform / phenomenon / track / event / sample)
4. What datastreams? (per system × per observed property × per result schema)
5. Units, observed properties, result schema per stream.
6. Mode: historical / live / batch / replayed.
7. Commands: now / later / never.

Output: filled-in worksheet that maps 1:1 to the CSAPI resources the integrator must POST.

### B. Golden-path compose stack (`compose/cs-go-golden-path/`)

- `docker-compose.yml` bringing up CS GO + Postgres+PostGIS.
- Seed script that creates one System, one Deployment, one Procedure, one Sampling Feature, one Datastream.
- README with curl smoke commands.
- Disk + memory + port footprint stated.
- Air-gap notes: which images need to be pre-pulled.

### C. Smoke-test script (`validators/smoke-test/`)

- Single bash script (also a Python equivalent) that hits an arbitrary CSAPI base URL and verifies:
  - landing page presence and required link rels;
  - `/conformance` exposes expected Part 1 + Part 2 URIs;
  - `/systems`, `/datastreams`, `/observations` reachable;
  - one observation roundtrip (POST → GET);
  - CORS headers (browser-client viability);
  - link integrity on returned documents.
- Exit code: 0 = pass, non-zero with summary table on fail.
- No internet access required at runtime.

### D. Three example mappings (`examples/`)

One worksheet, one publisher, one expected JSON tree per example. Examples chosen to span the modeling space:

- `weather-station-array/` — fixed deployment, multi-property, batch + live.
- `acoustic-sensor-array/` — multi-platform, event-driven, sampling-feature-heavy.
- `mobile-platform-gps-telemetry/` — moving system, time-series geometry.

### E. "Choose your server" matrix (`docs/guide/04-choose-your-server.md`)

Three rows, audit-grounded:

| Server | License | When to pick | When NOT to pick |
|---|---|---|---|
| **CS GO** (`SomethingCreativeStudios/connected-systems-go`) | MIT | Greenfield CSAPI-only deployment; want a single static binary; Postgres available | Need OSH driver library; need OSGi modularity |
| **OpenSensorHub** (`opensensorhub/osh-core`) | MPL-2.0 | Already have OSH or need extensive sensor-driver coverage | Container-only deployment; small footprint |
| **52°N pygeoapi-csapi** (`52North/connected-systems-pygeoapi`) | Apache-2.0 | Research / academic / Python-shop pilot | Production. Self-described "Proof of Concept." |

**Explicitly excluded from the matrix and why:** HAKUNAPI (OGC API Features only — not CSAPI; documentation defect on os4csapi.github.io tracked separately).

## 7. Honest disclaimers v0.1 must surface

These go in `START-HERE.md` and `08-what-v0.1-does-not-cover.md`:

1. **Part 2 has no OGC certification path yet.** ETS covers Part 1. Servers may *claim* Part 2 conformance; no third-party test confirms it.
2. **CS GO README cites Part 2 as "OGC 24-008."** Likely a typo for OGC 23-002. Filed upstream.
3. **Live interop bugs exist between `OS4CSAPI/ogc-client` and the OSH demo** (4 documented in `nsnarayanam/ogc-csapi-agent`'s README). Smoke tests will surface these.
4. **52°North demo `/conformance` is missing CSAPI URIs** as of audit date.
5. **No air-gap / FIPS / RHEL story in v0.1.** Defense / IC adoption requires v0.2 defense track.

## 8. Acceptance criteria for v0.1 release

A reviewer with a clean laptop, Docker, and `git` can:

1. Clone the repo, read `START-HERE.md`, and reach the "choose your path" decision in < 10 minutes.
2. Run `compose/cs-go-golden-path/` and see a healthy `/conformance` response.
3. Pick one example (e.g. weather-station-array), run its publisher, and see observations land.
4. Run `validators/smoke-test/` against the running stack and get exit 0.
5. Open `ogc-csapi-explorer` (separate clone) pointed at the stack and visually confirm the resources.

Total elapsed time on a representative laptop: **≤ 1 afternoon**.

## 9. Release process

- **Source of truth:** this repo, `main` branch.
- **Release artifact:** GitHub Release zip built by Action on tag `v0.1.0`.
- **Tag cadence:** v0.1 is one tag. v0.2 will be the defense / Part 3 / binary-encoding track.
- **Versioning:** semver. Pre-1.0 means breaking changes allowed; will be called out in RELEASE-NOTES.
- **License:** Apache-2.0 for the SDK itself (compatible with bundling MIT/BSD-3/MPL-2.0 components).
